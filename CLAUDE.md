## What this is

Academic software-reuse project (PJBL). Inventory system ("Stockys"). The grading focus is **code reuse** and **design patterns**, not features — most files carry Portuguese comments naming the reuse/pattern they demonstrate. Preserve those comments and the patterns when editing.

- Backend: Node 20 + TypeScript (ESM), Express, raw `pg` against PostgreSQL. Run via `tsx` (no build step).
- Frontend: React 18 + Vite + react-router-dom.
- Code, comments, and identifiers are in Portuguese (no accents).

## Commands


No test suite, no linter configured. Frontend `/api` is proxied to `http://localhost:3001` (see `frontend/vite.config.ts`).

## LPS variability (Software Product Line) — central to this project

The system ships as three editions controlled by `backend/.env`. Change `EDITION` (`BASIC` | `PLUS` | `ENTERPRISE`) and toggle `FEATURE_RELATORIOS` / `FEATURE_USUARIOS`, then restart backend and refresh the browser.

- Backend reads it once in `backend/src/config.ts` (Singleton) and exposes it at `GET /api/config`.
- Frontend `App.tsx` fetches `/api/config` on mount and shows/hides nav links + routes accordingly:
  - BASIC: Produtos, Categorias
  - PLUS: + Fornecedores, Movimentacoes, Usuarios, Relatorios
  - ENTERPRISE: + Produtos Pereciveis, Produtos Eletronicos
- When adding a feature, wire variability in **both** places: edition gating in `App.tsx` and (if applicable) a feature flag in `config.ts`.

## Architecture / reuse mechanisms

- **Generic CRUD factory** (`backend/src/routes/crud.ts`): `rotasCRUD(dao)` builds list/insert/update/delete Express routes for any DAO implementing the `DAO<T>` interface. All 5 entities (usuarios, categorias, fornecedores, produtos, movimentacoes) are mounted from this one factory in `server.ts`. Postgres error codes are mapped to friendly 400 messages here. Add a new entity = new DAO in `backend/src/dao/` + one `app.use(...)` line.
- **Frontend mirror**: `frontend/src/api.ts` exposes 4 generic helpers (`listar/inserir/atualizar/excluir`) used by every page; `components/TabelaCRUD.tsx` is the shared table UI. Pages in `frontend/src/pages/` are thin wrappers over it.
- **DAOs** in `backend/src/dao/` hold all SQL (parameterized). Shared pool in `backend/src/db.ts`. Types in `backend/src/models/index.ts`, mirrored in `api.ts`.

## ORM layer (Prisma) — side by side with raw `pg`

Added for the ORM / Database First assignment. The raw `pg` DAOs (`/api/*`) stay untouched; a parallel Prisma layer serves the same entities at `/api/orm/*`.

- **Database First**: `backend/prisma/schema.prisma` is generated by `npx prisma db pull` (introspects the live DB), then `npx prisma generate` builds the typed client. Do not hand-edit the schema as the source of truth — the DB (`sql/schema.sql`) is. `npx prisma studio` gives an auto-generated CRUD admin UI.
- `backend/src/orm/prisma.ts` — PrismaClient singleton. `backend/src/orm/routesORM.ts` — generic CRUD factory over a Prisma delegate (mirrors `routes/crud.ts`) plus relational `include` endpoints (`/api/orm/produtos-completos`, `/api/orm/pedidos-completos`). Mounted in `server.ts`.
- Needs `DATABASE_URL` in `backend/.env`.
- Schema grew from 5 to **9 tables** (added `produto_perecivel`, `produto_eletronico`, `pedido`, `pedido_item`) so there are 9 ORM models. Deliverables in `entrega/`.

## Design patterns (in `backend/src/patterns/`) — demonstrated, keep the structure

- **Singleton**: `config.ts` (`Config.get()`).
- **Template Method** (`patterns/template/`): abstract base defines the skeleton, subclasses fill steps.
  - `relatorio.ts` — `Relatorio` + `RelatorioEstoqueBaixo/Inventario/Movimentacao`, served at `GET /api/relatorios/:tipo`.
  - `importador.ts` (CSV/TSV), `exportador.ts`.
- **Strategy** (`patterns/strategy/`): interface + interchangeable implementations — `precoVenda.ts` (MargemFixa/Markup), `ordenacao.ts` (por nome/quantidade), `baixaEstoque.ts` (FIFO/LIFO).
